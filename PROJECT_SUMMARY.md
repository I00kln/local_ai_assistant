### 2.2 三层记忆架构

**模块名称**：`memory_manager.py` + `vector_store.py` + `sqlite_store.py`

**功能完成清单**：

- [x] L1 内存层（25条当前对话缓存）
- [x] L2 向量库层（ChromaDB + ONNX 嵌入）
- [x] L3 数据库层（SQLite 持久化）
- [x] L1→L2 自动溢出
- [x] L2→L3 定期归档
- [x] L3→L2 回填机制（冷数据激活）

- [x] 版本号并发控制

**关键技术决策**：

| 决策       | 理由                  |
| -------- | ------------------- |
| 三层分离     | 热数据快速访问、冷数据低成本存储    |
| ChromaDB | 轻量级向量库、本地持久化、无需外部服务 |
| ONNX 嵌入  | 本地推理、无网络依赖、隐私保护     |
| SQLite   | 嵌入式数据库、零配置、事务支持     |

```
│  │  内存层  │───▶│   向量库层   │───▶│   数据库层   │         │

│       │               │                   │                 │
```

**挑战与解决方案**：

> **挑战**：L2 向量库与 L3 数据库之间的数据同步问题\

> **挑战**：并发检索与迁移冲突\
> **方案**：迁移锁 + 版本号检测，检索时等待迁移完成

- [x] 相似度阈值过滤
- [x] 结果去重
- [x] 多样性控制
- [x] 本地/云端不同阈值

**检索策略**：

```
用户查询
    │
    ▼
┌─────────────────────────────────────────────────────────────┐
│  L1 内存层检索                                                │
│  • 关键词匹配（快速）                                          │
│  • 最近25条对话                                               │
│  • 返回：2-5条                                                │
└─────────────────────────────────────────────────────────────┘
    │
    ▼ 相似度 < 阈值？
┌─────────────────────────────────────────────────────────────┐
│  L2 向量库检索                                                │
│  • 向量相似度（语义）                                          │
│  • ChromaDB 查询                                              │
│  • 返回：top_k 条                                             │
└─────────────────────────────────────────────────────────────┘
    │
    ▼ 结果不足？
┌─────────────────────────────────────────────────────────────┐
│  L3 数据库检索                                                │
│  • 全文搜索（SQLite FTS）                                     │
│  • 冷数据回填到 L2                                            │
│  • 返回：补充结果                                              │
└─────────────────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────────────────┐
│  结果合并与排序                                               │
│  • 去重（相似度 > 0.95）                                      │
│  • 权重计算：similarity × time_decay × access_boost          │
│  • 多样性过滤                                                 │
└─────────────────────────────────────────────────────────────┘
    │
    ▼
  返回 top_k 结果
```

**关键技术决策**：

| 决策   | 理由              |
| ---- | --------------- |
| 多级检索 | 平衡速度与召回率        |
| 时间衰减 | 优先返回近期记忆        |
| 不同阈值 | 本地模型能力有限，需要更高阈值 |

**模块名称**：`async_processor.py` + `memory_archiver.py`

**功能完成清单**：

- [x] 异步处理队列

- [x] L2↔L3 双向流动
- [x] 熔断器保护

**生命周期流程**：

```
新对话产生
    │
    ▼
┌─────────────────────────────────────────────────────────────┐
│  异步处理队列                                                 │
│  • 最大队列：1000 条                                          │
│  • 溢出策略：丢弃最旧                                          │
│  • 批量处理：5 条/批                                           │
└─────────────────────────────────────────────────────────────┘
    │
    ├──────────────────┬──────────────────┐
    ▼                  ▼                  ▼
┌─────────┐      ┌─────────┐       ┌─────────┐
│  去重    │      │  压缩    │       │  分类    │
│ 哈希检测 │      │ LLM驱动  │       │ 存储策略 │
└─────────┘      └─────────┘       └─────────┘
    │                  │                  │
    └──────────────────┴──────────────────┘
                       │
                       ▼
              ┌─────────────────┐
              │   写入 L2/L3     │
              └─────────────────┘
                       │
        ┌──────────────┼──────────────┐
        ▼              ▼              ▼
   ┌─────────┐   ┌─────────┐   ┌─────────┐
   │ 热数据   │   │ 温数据   │   │ 冷数据   │
   │  L2     │   │ L2+L3   │   │  L3     │
   └─────────┘   └─────────┘   └─────────┘
        │              │              │
        │              │              │
        ▼              ▼              ▼
   定期访问        定期压缩        权重衰减
   权重提升        L2→L3迁移      自动遗忘
```

**压缩策略**：

| 策略     | 触发条件        | 压缩比     | 可用性        |
| ------ | ----------- | ------- | ---------- |
| LLM 压缩 | 文本 > 100 字符 | 30%-50% | 依赖 LLM 可用性 |
| 规则压缩   | LLM 不可用     | 60%-70% | 始终可用       |
| 高密度保护  | 检测到技术内容     | 100%    | 始终启用       |

**挑战与解决方案**：

> **挑战**：LLM 压缩失败导致记忆丢失\
> **方案**：实现熔断器，连续失败 5 次后暂停压缩，降级到规则压缩

> **挑战**：压缩耗时影响用户体验\
> **方案**：完全异步处理，用户无感知

### 2.5 废话过滤器

**模块名称**：`nonsense_filter.py`

**功能完成清单**：

- [x] 第一层：规则过滤（正则匹配）
- [x] 第二层：信息密度评分
- [x] 第三层：向量相似度匹配
- [x] 保护机制（纠错、数值、情感）
- [x] 高密度内容识别
- [x] 三级存储策略（full/sqlite\_only/discard）
- [x] 可配置废话库

**三层过滤架构**：

```
输入文本
    │
    ▼
┌─────────────────────────────────────────────────────────────┐
│  第一层：规则过滤（~1ms）                                     │
│  ─────────────────────────────────────────────────────────  │
│  • 单字回复："好"、"嗯"、"哦"                                 │
│  • 礼貌用语："谢谢"、"不客气"                                 │
│  • 确认回复："好的"、"明白了"                                 │
│                                                              │
│  输出：is_nonsense, confidence                               │
└─────────────────────────────────────────────────────────────┘
    │
    ▼ confidence < 0.8 ?
┌─────────────────────────────────────────────────────────────┐
│  第二层：信息密度评分（~5ms）                                 │
│  ─────────────────────────────────────────────────────────  │
│  • 计算语义向量                                               │
│  • 与历史记忆对比密度                                         │
│  • 低密度 = 信息量少                                          │
│                                                              │
│  输出：density_score, storage_type                           │
└─────────────────────────────────────────────────────────────┘
    │
    ▼ density < 0.3 ?
┌─────────────────────────────────────────────────────────────┐
│  第三层：向量匹配（~10ms）                                    │
│  ─────────────────────────────────────────────────────────  │
│  • 与废话库向量对比                                           │
│  • 相似度 > 0.9 判定为废话                                    │
│                                                              │
│  输出：final_decision, storage_type                          │
└─────────────────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────────────────┐
│  存储决策                                                     │
│  ─────────────────────────────────────────────────────────  │
│  • full: 存入向量库 + SQLite（有价值记忆）                    │
│  • sqlite_only: 仅存 SQLite（保持对话完整性）                 │
│  • discard: 完全丢弃（纯噪音）                                │
└─────────────────────────────────────────────────────────────┘
```

**保护机制**：

```python
PROTECTED_PATTERNS = [
    r"不[，,]?\s*是",           # 纠错："不，是100"
    r"更正[：:，,]?\s*",        # 更正："更正：..."
    r"\d+(?:元|美元|块|万|%|％)",  # 有意义的数字
    r"太[棒好美赞酷厉害]了",    # 情感反馈
    r"重要[：:]?",             # 重要信息
]
```

**挑战与解决方案**：

> **挑战**：误杀有价值的简短回复（如纠错、数值确认）\
> **方案**：实现保护模式列表，匹配保护模式的文本不会被判定为废话

**模块名称**：`memory_transaction.py`

**功能完成清单**：

- [x] 两阶段提交协议

**两阶段提交流程**：

```
┌─────────────────────────────────────────────────────────────┐
│  阶段一：准备（Prepare）                                      │
│  ─────────────────────────────────────────────────────────  │
│  1. 生成事务 ID                                               │
│  2. 在 SQLite 记录事务状态（PENDING）                         │
│  3. 执行 SQLite 写入                                          │
│  4. 更新状态为 PREPARING                                      │
└─────────────────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────────────────┐
│  阶段二：提交（Commit）                                       │
│  ─────────────────────────────────────────────────────────  │
│  1. 执行 ChromaDB 写入                                        │
│  2. 成功：更新状态为 COMMITTED                                │
│  3. 失败：回滚 SQLite，更新状态为 ROLLED_BACK                 │
└─────────────────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────────────────┐
│  恢复机制（启动时执行）                                        │
│  ─────────────────────────────────────────────────────────  │
│  1. 扫描 transactions 表                                      │
│  2. 查找 PENDING/PREPARING 状态的事务                         │
│  3. 重试未完成的事务                                          │
│  4. 清理过期事务（> 24h）                                     │
└─────────────────────────────────────────────────────────────┘
```

**挑战与解决方案**：

> **挑战**：程序崩溃导致 SQLite 和 ChromaDB 数据不一致\
> **方案**：事务状态持久化 + 启动时恢复，确保最终一致性

## 6. 经验总结

### 6.1 技术收获

**1. 三层记忆架构设计**

> 通过 L1/L2/L3 分层，成功平衡了访问速度与存储成本。L1 内存保证实时性，L2 向量库提供语义检索，L3 数据库确保持久化。这种设计在保证性能的同时，实现了记忆的长期保存。

**2. 事务协调器实现**

> 实现了 SQLite + ChromaDB 的两阶段提交，解决了跨存储系统的数据一致性问题。通过事务状态持久化，实现了崩溃恢复能力，这是构建可靠系统的关键基础设施。

**3. 熔断器模式应用**

> 在 LLM 压缩场景中引入熔断器，连续失败后自动降级到规则压缩。这种模式有效防止了级联失败，提升了系统稳定性。

**4. 事件驱动架构**

> 通过 EventBus 实现模块间解耦，MemoryManager 订阅 MEMORY\_WRITTEN 事件更新版本号。这种设计使得系统易于扩展，新增功能只需订阅相关事件。

**5. 结构化日志实践**

> 使用结构化日志记录关键操作，包含 event、关键字段、决策路径。这种日志格式便于问题定位和性能分析，是可观测性的重要组成部分。

### 6.2 踩过的坑

**案例一：并发检索与迁移冲突**

```
问题现象：
用户快速连续提问时，偶尔检索到重复或缺失的记忆

根因分析：
检索过程中，后台迁移线程正在将记忆从 L2 移动到 L3
导致检索结果不一致

解决方案：
1. 引入迁移锁，迁移时阻塞检索
2. 使用版本号检测并发修改
3. 检测到修改时自动重试（最多 3 次）

代码示例：
def search(self, query):
    if self._tx_coordinator.is_migration_active():
        self._tx_coordinator.wait_for_migration(timeout=5.0)
    
    for attempt in range(3):
        start_version = self._write_version
        results = self._do_search(query)
        if self._write_version == start_version:
            return results
        self.stats["retry_reads"] += 1
```

**案例二：废话过滤器误杀纠错信息**

```
问题现象：
用户发送"不，是100"被判定为废话并丢弃

根因分析：
"不"字开头的短文本被规则过滤拦截

解决方案：
实现保护模式列表，匹配保护模式的文本不会被判定为废话

保护模式示例：
- "不[，,]?\s*是"     → 纠错
- "\d+元"            → 价格
- "太[棒好]了"       → 情感反馈
```

**案例三：嵌入模型内存泄漏**

```
问题现象：
长时间运行后内存占用持续增长

根因分析：
嵌入缓存无上限，导致缓存无限增长

解决方案：
使用 LRU 缓存，限制最大缓存条目数

优化代码：
from functools import lru_cache

@lru_cache(maxsize=1000)
def embed_cached(self, text: str) -> Tuple[float, ...]:
    return self._embed_uncached(text)
```

**3. 观察者模式（事件总线）**

```python
class EventBus:
    def __init__(self):
        self._subscribers: Dict[EventType, List[Callable]] = defaultdict(list)
    
    def subscribe(self, event_type: EventType, handler: Callable):
        self._subscribers[event_type].append(handler)
    
    def publish(self, event_type: EventType, data: Any = None):
        for handler in self._subscribers[event_type]:
            try:
                handler(Event(event_type, data))
            except Exception as e:
                logger.error(f"Event handler failed: {e}")
```
