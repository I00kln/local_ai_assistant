### 2.3 后台任务调度器（新增）

**模块名称**：`background_scheduler.py`

**功能完成清单**：

- [x] 统一协调所有记忆状态变更操作
- [x] 任务优先级队列
- [x] 迁移锁保护（读写锁）
- [x] 读锁超时降级
- [x] 任务执行统计

**解决的问题**：

```
问题：并发竞态
├─ 检索与迁移同时进行，读取不一致数据
├─ 多个后台任务同时修改记忆状态
└─ 压缩/合并/遗忘操作相互干扰

解决方案：统一调度器
├─ 所有写操作通过调度器提交
├─ 迁移锁保护，检索等待迁移完成
├─ 任务队列串行化执行
└─ 读锁超时时降级为 L1 搜索
```

**任务类型**：

| 任务类型 | 说明 | 优先级 |
|----------|------|--------|
| MIGRATION_L2_TO_L3 | L2→L3 迁移 | HIGH |
| MIGRATION_L3_TO_L2 | L3→L2 回填 | HIGH |
| COMPRESSION | 记忆压缩 | NORMAL |
| MERGE | 记忆合并 | NORMAL |
| DEDUP | 去重 | NORMAL |
| DECAY | 权重衰减 | LOW |
| FORGET | 遗忘 | LOW |

### 2.4 生命周期管理器（新增）

**模块名称**：`lifecycle_manager.py`

**功能完成清单**：

- [x] 集中注册服务清理函数
- [x] 按优先级顺序关闭服务
- [x] 超时保护，避免无限等待
- [x] 发布 SHUTDOWN 事件通知
- [x] 强制停止支持

**解决的问题**：

```
问题：关闭卡顿
├─ 每个线程 join 5秒，多个线程累加
├─ 关闭顺序混乱，依赖关系错误
└─ 硬编码清理，难以维护

解决方案：生命周期管理器
├─ 按优先级顺序关闭（CRITICAL → BACKGROUND）
├─ 并行关闭独立服务
├─ 超时保护，强制停止
└─ 统一注册，易于维护
```

**服务关闭优先级**：

| 优先级 | 服务 | 说明 |
|--------|------|------|
| CRITICAL (100) | thread_pool_manager | 先停止接受新任务 |
| HIGH (80) | event_bus, async_processor | 停止事件分发 |
| NORMAL (50) | memory_manager, vector_store | 刷写缓冲区 |
| LOW (20) | sqlite_store | 关闭数据库连接 |
| BACKGROUND (10) | health_check | 停止监控 |

### 2.5 线程池管理器（新增）

**模块名称**：`thread_pool_manager.py`

**功能完成清单**：

- [x] 按任务类型隔离线程池
- [x] 拒绝策略保护
- [x] 统一监控和统计
- [x] 优雅关闭支持

**解决的问题**：

```
问题：线程爆炸
├─ 每个模块创建自己的线程
├─ 无限制创建导致资源耗尽
└─ 任务堆积无法控制

解决方案：统一线程池管理器
├─ IO 任务：8 线程池
├─ CPU 任务：4 线程池
├─ UI 任务：2 线程池
└─ 拒绝策略：REJECT / DROP_OLDEST / CALLER_RUNS
```

```
┌─────────────────────────────────────────────────────────────────────┐
│                        两阶段提交流程                                 │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  Phase 1: PREPARE                                                    │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  1. 生成事务 ID                                               │   │
│  │  2. 在 SQLite 记录事务状态（PENDING）                          │   │

│  │  4. 更新事务状态（PREPARED）                                   │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                              │                                       │
│                              ▼                                       │
│  Phase 2: COMMIT                                                     │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  1. 执行 ChromaDB 操作                                        │   │
│  │  2. [成功] 更新事务状态（COMMITTED）                           │   │
│  │  3. [失败] 回滚 SQLite 操作，更新状态（ROLLED_BACK）           │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                      │
│  Recovery: 启动时检查 PENDING/PREPARED 事务并重试                    │
│                                                                      │

```

## 6. 经验总结

### 6.1 成功经验

1. **统一调度器解决并发问题**
   - 所有写操作通过调度器提交
   - 迁移锁保护，避免竞态条件
   - 读锁超时降级，保证可用性

2. **生命周期管理解决关闭问题**
   - 按优先级顺序关闭
   - 超时保护，避免卡顿
   - 统一注册，易于维护

3. **线程池管理解决资源问题**
   - 按任务类型隔离
   - 拒绝策略保护
   - 统一监控统计

### 6.2 遇到的挑战

1. **并发竞态问题**
   - 问题：检索与迁移同时进行
   - 方案：引入后台任务调度器，统一协调

2. **关闭卡顿问题**
   - 问题：多个线程 join 累加超时
   - 方案：生命周期管理器，按优先级关闭

3. **线程爆炸问题**
   - 问题：每个模块创建自己的线程
   - 方案：统一线程池管理器，隔离控制
