# AI Coding Agent 可执行指令集 

> 本指令集遵循 **优先级驱动、输出结构化、自动校验** 原则。  
> 规则冲突时，按下方优先级裁决。所有输出必须遵循固定格式。

---

## 规则优先级（从高到低）

1. **安全性** — 输入校验、沙箱、资源配额  
2. **正确性** — 逻辑闭环、类型安全、原子性回滚  
3. **稳定性** — 超时、降级、断路器、熔断  
4. **可维护性** — SOLID、接口隔离、依赖倒置  
5. **简洁性** — DRY、函数长度、嵌套深度

---

## 核心工程规范（精简必选）

### 1. 防御与容错
- **波斯特尔定律**：发送严格，接收宽容  
- **所有异步调用**：必须设置 `timeout` 和 `.catch(fallback)`  
- **所有 if**：必须配套 `else`（或显式注释“无动作”）  
- **所有 switch**：必须包含 `default`  
- **复杂操作**：失败时必须回滚到检查点（原子性）

### 2. 安全与资源
- **输入洗涤**：任何外部来源数据（API、用户输入、数据库）必须经过 JSON Schema 或类型守卫校验  
- **循环/递归限制**：显式设置最大迭代次数（如 `for (let i=0; i<max; i++)`）  
- **禁止直接调用系统 API**：必须通过 `Wrapper` 层访问文件/网络

### 3. 代码规模红线（触发自动重构）
满足以下 **任一条件** 时必须拆分/重构：
- 函数体超过 **30 行**
- 参数数量超过 **4 个**
- 嵌套深度（if/for/while）≥ **3 层**
- 相同逻辑出现 ≥ **2 次**

### 4. 错误处理标准化
- **禁止** `throw new Error("...")`  
- **必须** 使用自定义错误类，包含 `code` 字段且可序列化：

```typescript
class BusinessError extends Error {
  constructor(public code: string, message: string) {
    super(message);
    this.name = "BusinessError";
  }
}
// 使用示例：throw new BusinessError("INSUFFICIENT_BALANCE", "余额不足")
```

### 5. 策略模式替代深层嵌套
- 当 `if-else` 或 `switch` 分支 ≥ 3 个时，强制使用策略注册表：

```typescript
interface Strategy<T> {
  execute(input: T): T;
}

class StrategyRegistry<T> {
  private map = new Map<string, Strategy<T>>();
  register(type: string, strategy: Strategy<T>) { this.map.set(type, strategy); }
  get(type: string): Strategy<T> {
    return this.map.get(type) ?? { execute: (i) => i }; // 默认透传
  }
}
```

### 6. 结构化日志
- **禁止** `console.log("xxx")`  
- **必须** 使用结构化日志，包含 `event`、关键字段、决策路径：

```typescript
logger.info({
  event: "PAYMENT_GATEWAY_FALLBACK",
  userId: "123",
  primaryAttempt: "failed",
  fallbackUsed: "localCache"
});
```

### 7. 不可变与类型
- 核心业务数据流使用 `Readonly<T>` 或 `Immutable` 结构  
- **严禁** 使用 `any` 或 `Object` 作为类型  
- 函数参数如需修改，必须明确返回新副本（不修改入参）

## 架构约束

- **单向依赖**：模块间禁止循环引用。上层可依赖下层，同层仅通过接口或事件通信。  
- **纯净领域**：Domain 层不包含任何框架（Express/Spring）或数据库（SQL）代码。  
- **接口隔离**：不设计上帝接口，每个接口只包含调用方需要的方法。  
- **迪米特法则**：禁止链式调用超过一级（如 `user.address.city` 允许，`user.getAddress().getCity()` 禁止）。

---

## 自动重构与质量门禁

在每次生成代码后，必须执行以下检查并输出结果：

| 检查项 | 通过条件 | 失败动作 |
|--------|----------|----------|
| 函数行数 | ≤ 30 | 拆分为多个函数 |
| 参数个数 | ≤ 4 | 封装为对象参数 |
| 嵌套深度 | ≤ 2 | 提取子函数或策略模式 |
| 重复代码 | 无相同逻辑片段 | 抽取为公共函数 |
| 类型覆盖 | 无 any / 隐式 any | 显式声明类型 |

## 冲突解决示例

| 冲突 | 按优先级裁决 |
|------|--------------|
| OCP（扩展） vs 快速修复（修改基类） | 优先遵守 OCP，不得修改已验证基类 |
| DRY（抽象） vs 可读性（内联） | 若抽象后复杂度降低则抽象，否则保持内联但添加注释 |
| 防御式设计（大量检查） vs 简洁性 | 安全性优先，但可将重复检查抽取为守卫函数 |

## 最终指令

1. 每次输出必须包含 `<design>`、`<plan>`、`<code>`、`<check>` 四个部分。  
2. `<check>` 中的每一项必须明确标记 `[x]` 或解释为何不适用。  
3. 任何违反“核心工程规范”的代码将被拒绝。  
4. 遇到模糊需求时，优先选择 **安全性 > 正确性 > 稳定性** 的实现路径。
